# JSON API Data Explained - Documentation

All data is obtained via:

  • The Solana RPC interface
  • Solana command line programs
  • Jito on-chain Stakenet CLI
  • Vortex | Validator Explorer for Vote Latency Data
  • Shinobi Systems' stakeview.app voting performance API
  • Various Firedancer validators' GUI websocket metrics for slot durations as observed by each validator
    (note: we are polling a geo-diverse group of metrics and using only the smallest observed value for each slot's duration. We also ignore any skipped slots and the block immediately after a skipped slot)

# Subscribing to Updates

# About the JSON Keypairs

Most of the JSON keypair names should be self-explanatory for a Solana validator. Here are some explanations:

  • epoch_credits: The number of vote credits in the current epoch. Prior to TVC this was nearly the same as votes_cast. Now, with TVC, epoch_credits receives a higher score for low latency votes
  • leader_slots: The number of slots assigned to a validator where it is responsible to produced blocks. These are assigned in groups of 4
  • mev_earned: Total MEV where some or all may go to the validator or some or all may go to the validator's stake holders
  • mev_to_validator: The amount of mev_earned that goes to the validator based on the validator's mev_commission
  • rewards: The total_block_rewards_before_burn is the sum of (signatures * 5000 lamports) + priority_fees and shown in SOL
    As of 2024 half of block rewards are "burned" so we have validator_signature_fees = 0.5 * (signatures * 5000 lamports) and validator_priority_fees going to the validator and both are shown in SOL. The sum of these two numbers is shown in SOL as total_block_rewards_after_burn or rewards
  • signatures: The number of shreds and blocks the validator signed as part of consensus
  • skip_rate: The percentage of slots where a validator missed producing a block during a leader slot
  • stake_percentage: The percentage of this validator's active stake compared to the total active stake
  • vote_cost: The cost of participating in consensus where a validator is charged 5000 lamports for each vote cast
  • votes_cast: The number of blocks the validator voted on for consensus

# The API URLs Explained

Validator Rewards Data

  1. Validator Data

    https://api.trillium.so/validator_rewards/

    Returns the latest epoch's rewards data for all validators

    https://api.trillium.so/validator_rewards

    If an epoch_number is provided, returns that epoch's data for all validators

    https://api.trillium.so/validator_rewards/633

    If an identity_pubkey is provided, returns the last 10 epochs of data for that validator

    https://api.trillium.so/validator_rewards/tri1cHBy47fPyhCvrCf6FnR7Mz6XdSoSBah2FsZVQeT

    If a vote_account_pubkey is provided, returns the last 10 epochs of data for that validator

  2. Validator Rewards : Averages over Last 10 Epochs

    https://api.trillium.so/ten_epoch_validator_rewards

    Returns averages for most numerical data over the last 10 epochs for all validators

    The passing of a pubkey parameter works the same as above

  3. Validator Rewards : Weighted Averages over Last 10 Epochs

    https://api.trillium.so/recency_weighted_average_validator_rewards

    Returns epoch weighted averages for most numerical data over the last 10 epochs for all validators

    The passing of a pubkey parameter works the same as above

    Averaging is performed using weighted values from the prior 10 epochs. However, epochs are not weighted equally. Instead, the following weighting1 is used for each epoch to determine that epoch's contribution to each validator's score:

    Current Epoch : 0.2649
Epoch - 1     : 0.1987
Epoch - 2     : 0.1490
Epoch - 3     : 0.1118
Epoch - 4     : 0.0838
Epoch - 5     : 0.0629
Epoch - 6     : 0.0471
Epoch - 7     : 0.0354
Epoch - 8     : 0.0265
Epoch - 9     : 0.0199

    1 Note: this epoch weighting idea and weight values are from Shinobi Systems xshin.fi

    Older epochs contribute less total scoring weight than newer epochs. The current epoch contributes a little more than a quarter of the overall score, and the current and prior three epochs contribute almost 3/4 of total score.

Epoch Data

  1. Epoch Rewards : Last 10 Epochs

    https://api.trillium.so/epoch_data/

    Returns epoch rewards data for the last 10 epochs

    https://api.trillium.so/epoch_data/633

    If an epoch_number is provided, returns only that epoch's data

  2. Epoch Rewards : Averages over Last 10 Epochs

    https://api.trillium.so/ten_epoch_aggregate_data

    Returns averages for most numerical data over the last 10 epochs at the epoch level

  3. Skipped Slots Analysis

    https://api.trillium.so/skip_analysis/

    Returns Skipped slot analysis for the current epoch by validator

    https://api.trillium.so/skip_analysis/699

    If an epoch_number is provided, returns that epoch's data for all validators

    https://api.trillium.so/skip_analysis/Awes4Tr6TX8JDzEhCZY2QVNimT6iD1zWHzf1vNyGvpLM

    If an identity_pubkey is provided, returns the last 10 epochs of data for that validator

# Usage & Attribution

Usage & Rate Limits

This API is open to the public and does not require any authentication, tokens, or registration.

While Trillium does not currently enforce rate limits, we kindly ask that you use the API responsibly. If we notice performance degradation due to increased usage, rate limits may be implemented in the future.

Attribution

If you are using a Trillium API for any non-personal use, we request that you provide attribution on your site or project.
